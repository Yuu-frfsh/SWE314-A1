================================================================================
                    EDUCATIONAL ENCRYPTION TOOL
                    USE CASE DOCUMENTATION
================================================================================

This document provides a comprehensive overview of all use cases in the system,
including their descriptions, actors, relationships, and flow descriptions.

================================================================================
                              ACTORS
================================================================================

Primary Actor:
  - User: The individual using the Educational Encryption Tool to learn about
         cryptography. The user interacts with the system through the console
         interface to perform encryption, decryption, and cryptanalysis operations.

================================================================================
                         USE CASE CATEGORIES
================================================================================

1. System Navigation Use Cases (UC9, UC10, UC21, UC22)
2. Encryption Use Cases (UC1-UC6)
3. Decryption Use Cases (UC7)
4. Cryptanalysis Use Cases (UC8)
5. Supporting/Utility Use Cases (UC11-UC20)

┌─────────────────────────────────────────────────────────────────────────────┐
│ UC1: Encrypt with Monoalphabetic Cipher                                     │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user encrypts plaintext using the Monoalphabetic Substitution Cipher
  (Caesar Cipher) with a shift key, and views step-by-step encryption process.

Preconditions:
  - System is running
  - User is at the main menu

Main Flow:
  1. User selects option "1" (Monoalphabetic Substitution)
  2. System includes UC13 (Get Plaintext Input)
  3. System includes UC15 (Get Key Input) - expects integer shift value
  4. System includes UC11 (Validate User Input)
  5. System includes UC12 (Display Step-by-Step Process)
  6. System performs Monoalphabetic encryption
  7. System includes UC16 (Display Encryption Result)
  8. System returns to main menu

Alternative Flows:
  2a. If plaintext is empty:
      - System displays error message
      - Returns to step 2
  3a. If key is invalid (not an integer):
      - System displays error message
      - Returns to step 3

Postconditions:
  - Plaintext has been encrypted
  - Ciphertext is displayed
  - Step-by-step process has been shown
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC13, UC15, UC16
  - Uses: MonoalphabeticCipher class

┌─────────────────────────────────────────────────────────────────────────────┐
│ UC3: Encrypt with Playfair Cipher                                           │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user encrypts plaintext using the Playfair Cipher with a keyword,
  and views step-by-step encryption process including 5x5 matrix generation.

Preconditions:
  - System is running
  - User is at the main menu

  1. User selects option "3" (Playfair Cipher)
  2. System includes UC13 (Get Plaintext Input)
  3. System includes UC15 (Get Key Input) - expects keyword string
  4. System includes UC11 (Validate User Input) - checks alphabetic only
  5. System builds 5x5 key matrix from keyword
  6. System prepares plaintext into digraphs
  7. System includes UC12 (Display Step-by-Step Process) - shows matrix and digraphs
  8. System performs Playfair encryption
  9. System includes UC16 (Display Encryption Result)
  10. System returns to main menu

Alternative Flows:
  2a. If plaintext is empty:
      - System displays error message
      - Returns to step 2
  3a. If key is empty or contains non-alphabetic characters:
      - System displays error message
      - Returns to step 3

Postconditions:
  - Plaintext has been encrypted
  - Ciphertext is displayed
  - Step-by-step process has been shown (including matrix)
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC13, UC15, UC16
  - Uses: PlayfairCipher class


┌─────────────────────────────────────────────────────────────────────────────┐
│ UC4: Encrypt with Keyed Transposition Cipher                                │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user encrypts plaintext using the Keyed Transposition Cipher with
  a 2x5 permutation matrix key, and views step-by-step encryption process.

Preconditions:

  - User is at the main menu

Main Flow:
  1. User selects option "4" (Keyed Transposition)
  2. System includes UC13 (Get Plaintext Input)
  3. System prompts for first row of key (plaintext indices)
  4. System prompts for second row of key (ciphertext positions)
  5. System includes UC11 (Validate User Input) - validates 5 values per row
  6. System includes UC12 (Display Step-by-Step Process) - shows 2x5 matrix
  7. System pads plaintext to multiple of 5 if needed
  8. System performs Keyed Transposition encryption
  9. System includes UC16 (Display Encryption Result)
  10. System returns to main menu

Alternative Flows:
  2a. If plaintext is empty:
      - System displays error message
      - Returns to step 2
  4a. If key format is invalid (not 5 values per row):
      - System displays error message
      - Returns to step 3

Postconditions:
  - Plaintext has been encrypted
  - Ciphertext is displayed
  - Step-by-step process has been shown
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC13, UC15, UC16
  - Uses: KeyedTranspositionCipher class


┌─────────────────────────────────────────────────────────────────────────────┐
│ UC5: Encrypt with Combined Cipher                                           │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user encrypts plaintext using a combination of Monoalphabetic Cipher
  and another cipher (Vigenere, Playfair, or Keyed Transposition), viewing
  step-by-step process for both algorithms.

Preconditions:
  - System is running
  - User is at the main menu

Main Flow:
  1. User selects option "5" (Combined Encryption)
  2. System includes UC20 (Choose Second Algorithm)
  3. System includes UC13 (Get Plaintext Input)
  4. System includes UC15 (Get Key Input) - gets Monoalphabetic shift key
  5. System includes UC11 (Validate User Input) - validates integer shift
  6. System includes UC15 (Get Key Input) - gets second algorithm key
     (format depends on selected algorithm)
  7. System includes UC11 (Validate User Input) - validates second key
  8. System includes UC12 (Display Step-by-Step Process) - shows both algorithms
  9. System performs Monoalphabetic encryption first
  10. System performs second algorithm encryption on intermediate result
  11. System includes UC16 (Display Encryption Result)
  12. System returns to main menu

Alternative Flows:
  2a. If user cancels algorithm selection:
      - Use case terminates
      - Returns to main menu
  3a. If plaintext is empty:
      - System displays error message
      - Returns to step 3
  5a. If Monoalphabetic key is invalid:
      - System displays error message
      - Returns to step 4
  6a. If second algorithm key is invalid:
      - System displays error message
      - Returns to step 6

Postconditions:
  - Plaintext has been encrypted with combined ciphers
  - Ciphertext is displayed
  - Step-by-step process has been shown for both algorithms
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC13, UC15, UC16, UC20
  - Uses: CombinedCipher, MonoalphabeticCipher, and selected second cipher class


┌─────────────────────────────────────────────────────────────────────────────┐
│ UC6: Encrypt with DES Cipher                                                 │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user encrypts plaintext using the simplified DES (Data Encryption Standard)
  cipher with a key, viewing detailed step-by-step encryption process.

Preconditions:
  - System is running
  - User is at the main menu

Main Flow:
  1. User selects option "6" (DES simplified)
  2. System includes UC13 (Get Plaintext Input)
  3. System includes UC15 (Get Key Input) - expects key string (first 8 chars used)
  4. System includes UC11 (Validate User Input)
  5. System converts plaintext to binary
  6. System builds DES key schedule
  7. System includes UC12 (Display Step-by-Step Process) - shows DES operations
  8. System performs DES encryption (block by block)
  9. System converts result to hexadecimal
  10. System includes UC16 (Display Encryption Result) - displays hex ciphertext
  11. System returns to main menu

Alternative Flows:
  2a. If plaintext is empty:
      - System displays error message
      - Returns to step 2
  3a. If key is empty:
      - System displays error message
      - Returns to step 3

Postconditions:
  - Plaintext has been encrypted using DES
  - Hexadecimal ciphertext is displayed
  - Step-by-step process has been shown
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC13, UC15, UC16
  - Uses: DESCipher class

┌─────────────────────────────────────────────────────────────────────────────┐
│ UC7: Decrypt Text                                                            │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user decrypts ciphertext by selecting an encryption algorithm and providing
  the appropriate key, viewing step-by-step decryption process.

Preconditions:
  - System is running
  - User is at the main menu
  - User has ciphertext to decrypt

Main Flow:
  1. User selects option "7" (Decrypt Text)
  2. System includes UC19 (Select Decryption Algorithm)
  3. System includes UC14 (Get Ciphertext Input)
  4. System determines key format based on selected algorithm
  5. System includes UC15 (Get Key Input) - format depends on algorithm
     (for Keyed Transposition: prompts for 2-line key)
  6. System includes UC11 (Validate User Input)
  7. System includes UC12 (Display Step-by-Step Process) - shows decryption steps
  8. System performs decryption using selected algorithm
  9. System includes UC17 (Display Decryption Result)
  10. System returns to main menu

Alternative Flows:
  2a. If user selects invalid algorithm:
      - System displays error message
      - Use case terminates
      - Returns to main menu
  3a. If ciphertext is empty:
      - System displays error message
      - Returns to step 3
  5a. If key is invalid or empty:
      - System displays error message
      - Returns to step 5

Special Cases:
  - For Keyed Transposition (option 4): Key input requires two separate lines
  - For DES (option 5): Ciphertext input is in hexadecimal format

Postconditions:
  - Ciphertext has been decrypted
  - Plaintext is displayed
  - Step-by-step process has been shown
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC12, UC14, UC15, UC17, UC19
  - Uses: MonoalphabeticCipher, VigenereCipher, PlayfairCipher,
          KeyedTranspositionCipher, or DESCipher class (depending on selection)

┌─────────────────────────────────────────────────────────────────────────────┐
│ UC8: Perform Frequency Analysis                                             │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: User
Type: Primary Use Case

Brief Description:
  The user performs frequency analysis on ciphertext to break monoalphabetic
  substitution ciphers. The system automatically detects the key and decrypts
  the ciphertext.

Preconditions:
  - System is running
  - User is at the main menu
  - User has ciphertext to analyze

Main Flow:
  1. User selects option "8" (Frequency Analysis)
  2. System includes UC14 (Get Ciphertext Input)
  3. System includes UC11 (Validate User Input)
  4. System counts letter frequencies in ciphertext
  5. System calculates percentage frequencies
  6. System displays frequency table (letter, count, percentage, expected %)
  7. System identifies most frequent letters
  8. System suggests possible letter substitutions
  9. System automatically detects encryption key using correlation analysis
  10. System automatically decrypts ciphertext using detected key
  11. System displays detected key and decrypted plaintext
  12. System returns to main menu

Alternative Flows:
  2a. If ciphertext is empty:
      - System displays error message
      - Returns to step 2
  9a. If key cannot be detected (low correlation, wrong cipher type, etc.):
      - System displays message indicating detection failure
      - Continues to step 12

Postconditions:
  - Frequency analysis has been performed
  - Letter frequencies have been displayed
  - Key has been detected (if possible)
  - Plaintext has been decrypted and displayed (if key detected)
  - User is returned to main menu

Related Use Cases:
  - Includes: UC10, UC11, UC14
  - Uses: FrequencyAnalysis class
  - Uses: MonoalphabeticCipher class (for auto-decryption)

┌─────────────────────────────────────────────────────────────────────────────┐
│ UC12: Display Step-by-Step Process                                          │
└─────────────────────────────────────────────────────────────────────────────┘

Actor: System
Type: Supporting Use Case (Included)

Brief Description:
  The system displays a detailed, educational step-by-step explanation of the
  encryption or decryption process being performed.

Preconditions:
  - Algorithm has been selected
  - Input text and key have been provided

Main Flow:
  1. System displays algorithm name and description
  2. System displays input text
  3. System displays key information
  4. For each step in the algorithm:
     a. System shows current state
     b. System shows transformation being applied
     c. System shows intermediate result
  5. System displays final result explanation

Algorithm-Specific Displays:
  - Monoalphabetic: Character mappings and shift values
  - Vigenere: Key alignment and shift calculations
  - Playfair: 5x5 matrix, digraph preparation, encryption rules
  - Keyed Transposition: 2x5 matrix and block transformations
  - DES: Binary operations, key schedule, S-box applications
  - Combined: Steps for both algorithms sequentially

Postconditions:
  - Step-by-step process has been displayed
  - User understands how the algorithm works

Related Use Cases:
  - Included by: UC1, UC2, UC3, UC4, UC5, UC6, UC7
  - Extended by: UC24 (Display Detailed Steps) - for more verbose output

================================================================================
                    NOTES ON PLANTUML DIAGRAMS
================================================================================

1. To generate diagrams from PlantUML code:
   - Use an online PlantUML editor (http://www.plantuml.com/plantuml/)
   - Use PlantUML plugin in IDE (IntelliJ, VS Code, etc.)
   - Use PlantUML command-line tool with Java

2. Diagram Types:
   - Complete Use Case Diagram: Shows all use cases and relationships
   - Simplified Primary: Shows only primary use cases grouped by category
   - Detailed Relationships: Focuses on includes, extends, generalization
   - Frequency Analysis Detail: Shows UC8 specific relationships
   - Decryption Detail: Shows UC7 specific relationships

3. Relationship Notation:
   - Association (-->): Direct interaction between actor and use case
   - Includes (..>): Mandatory inclusion, always executes
   - Extends (<..): Optional extension, conditionally executes
   - Generalization (<|--): Inheritance/specialization relationship

4. Customization:
   - Colors can be adjusted by modifying skinparam usecase sections
   - Themes can be changed (plain, cerulean, reddress-darkblue, etc.)
   - Packages can be added for better organization
   - Notes can be added for additional clarification
